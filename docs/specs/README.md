# 轻量 Spec-Driven 流程

该目录用于存放功能规格（spec）。本项目采用的是轻量 spec-driven，而不是重型设计流程。

## 目标

- 在实现前先固定关键语义，减少来回返工
- 让测试用例直接对应验收标准
- 保持原型开发速度，不把流程做重

## 什么时候必须写 spec

以下情况必须新增或更新 spec：

- 新功能
- 重要交互行为变化
- 错误语义变化
- 测试策略或验收标准明显变化

以下情况通常可以不单独写新 spec，但仍应检查现有 spec 是否需要更新：

- 纯文档修改
- 明显的拼写修复
- 不改变行为的小型重构

## 推荐流程

1. 在实现前创建或更新 `docs/specs/NNNN-short-name.md`
2. 使用 [`TEMPLATE.md`](./TEMPLATE.md) 填写最小必要内容
3. 按 spec 编写代码和测试
4. 执行 `make test`
5. 若实现语义与原 spec 不一致，先更新 spec，再视为完成

## 写法要求

- 优先一页以内
- 用行为和边界情况说清楚，不写空洞背景
- 每条验收标准都应该能落到测试或手工验证上
- 当需求还在探索中时，明确写出“假设”和“非目标”

## 文件命名

- 推荐：`0001-short-name.md`
- 编号不必追求严格连续，但要保持唯一
- 一个 spec 只描述一个清晰主题
