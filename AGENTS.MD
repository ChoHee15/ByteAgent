# Code Agent 开发核心准则 (Definition of Done)

在执行任何代码修改或添加新功能时，必须严格遵守以下工作流：

## 1. 测试同步 (Test Co-development)
- 任何核心逻辑的修改或新功能的添加，**必须**同步提供或更新对应的 `_test.go` 文件。
- Go 测试需严格采用表驱动测试 (Table-driven tests) 范式。
- 涉及外部系统（如系统命令执行、外部 API）必须使用 Mock，禁止在单元测试中发起真实网络请求或破坏性系统调用。
- 默认开发测试只允许运行 mock/本地测试层；真实大模型 API 测试仅允许放在显式的 integration 测试层中，并通过单独命令运行。

## 2. 回归验证 (Regression Guardrails)
- 在你认为代码编写完成后，**必须**主动使用你的 bash 执行能力运行命令：`make test`。
- 你必须读取测试的标准输出。如果有任何历史测试用例失败（即破坏了现有功能），你必须优先修复这些 Regression，然后再向用户报告任务完成。
- **严禁**在测试未全部通过的情况下宣称功能已开发完毕。
- 真实 API smoke test 仅在需要验证线上模型连通性时运行 `make test-integration`，不得将其当作默认回归命令。

## 3. 状态管理与文档维护 (State Management)

为了保证项目的长期可维护性，你必须在每次交互中严格遵守以下文档读写流程：

1. **工作前先看黑板 (`TODO.md`)**：
   - 在开始编写任何代码前，必须先读取 `TODO.md`，确认当前正在进行的任务目标，若没有对应的任务则创建相应的任务项目。
   - 当你完成了一个功能并通过了测试（`make test`）后，必须主动修改 `TODO.md`，将对应任务标记为已完成（如改为 `[x]`）。

2. **工作后写日志 (`CHANGELOG.md`)**：
   - 任何涉及功能添加、Bug 修复或重大的架构重构，在代码提交或回复用户“任务已完成”之前，必须在 `CHANGELOG.md` 中追加一条记录。
   - 记录需包含日期、简短的功能描述，以及覆盖了哪些边界情况。

## 4. 提交纪律 (Commit Discipline)

- 当某项功能已经完成并且测试/验证成功后，**不得**直接创建 commit。
- 你必须先向用户明确报告验证结果，并请求用户确认是否提交当前修改。
- 只有在得到用户确认后，才可以执行 `git add` / `git commit`。
- 提交信息必须使用 GitHub 社区常见的 Conventional Commits 风格，例如：`feat: ...`、`fix: ...`、`docs: ...`、`test: ...`、`refactor: ...`。
- 如需补充 commit body，应简洁说明本次修改内容、影响范围或关键边界情况，避免无信息量描述。
